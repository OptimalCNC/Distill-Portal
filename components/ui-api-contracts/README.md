## TypeScript Bindings

The frontend app consumes these contracts from TypeScript. To avoid a hand-maintained duplicate, the crate generates TypeScript type declarations directly from the Rust definitions and checks the output into `components/ui-api-contracts/bindings/`.

### Tool choice

The contracts crate uses [`ts-rs`](https://github.com/Aleph-Alpha/ts-rs) `12.0.1` (pinned in `Cargo.toml`) as the codegen tool.

Rationale, compared to the main alternative, [`typeshare`](https://github.com/1Password/typeshare):

- **`ts-rs` is derive-driven and stays inside the Rust toolchain.** A `cargo test` invocation regenerates the bindings; no external CLI install step is required. `typeshare` ships codegen as a separate `typeshare-cli` binary that contributors would need to install and keep on the right version, and the CI image would need to carry.
- **`ts-rs` understands the serde attributes we depend on.** `#[serde(rename_all = "snake_case")]` on enums is reflected — `Tool` emits `"claude_code" | "codex"`, matching the JSON wire shape. `#[serde(flatten)]` on `StoredSessionView.session` is reflected via the companion `#[ts(flatten)]` attribute, so the generated TS type inlines the `StoredSessionRecord` fields. `typeshare` has a narrower serde surface and historically has gaps around flattened structs, so for the attributes that matter to our contract shapes, `ts-rs` is the closer match.
- **Generated output is stable.** Each run of the generator produces byte-identical files, so the staleness check can compare against a fresh temp-dir regeneration with a simple `assert_eq!`.
- **The codegen dependency is kept out of the backend build.** `ts-rs` is listed as an *optional* dependency behind the `ts-bindings` feature, which is off by default. Backend and workspace builds never compile `ts-rs` unless a developer explicitly opts in.

`typeshare` remains a reasonable fallback if the project ever needs to emit Kotlin/Swift alongside TypeScript. For a single TS target, the in-tree, cargo-native ergonomics of `ts-rs` are the better match.

### Where the generated files live

```

├── Cargo.toml
├── README.md
├── src/lib.rs                 # Rust source of truth
├── bindings/                  # generated TypeScript (checked in)
│   ├── ImportReport.ts
│   ├── ImportSourceSessionsRequest.ts
│   ├── PersistedScanError.ts
│   ├── RescanReport.ts
│   ├── SessionSyncStatus.ts
│   ├── SourceSessionView.ts
│   ├── StoredSessionRecord.ts
│   ├── StoredSessionView.ts
│   └── Tool.ts
└── tests/ts_bindings.rs       # regenerator + staleness check
```

Each `.ts` file is owned by the codegen: do not edit by hand. The in-file banner (`// This file was generated by [ts-rs] ...`) is part of the ts-rs output, not added by this crate.

The helper function `source_key(tool, source_session_id)` in `src/lib.rs` is deliberately **not** exported to TypeScript. It is a Rust-side convenience for building the `"tool:source_session_id"` session-key format and is not part of any HTTP payload. The frontend MUST NOT reconstruct or mutate this string. Two distinct identity rules apply on the frontend:

- **Import identity** is always the backend-provided `SourceSessionView.session_key` field (already formatted as `"tool:source_session_id"` with a single colon by the Rust `source_key` helper). The unified-list join surfaces it as `SessionRow.sourceSessionKey` (`apps/frontend/src/features/sessions/types.ts:26-27`); reads happen at `apps/frontend/src/features/sessions/mergeSessions.ts:62` (the `presence === "both"` branch) and `apps/frontend/src/features/sessions/mergeSessions.ts:85` (the `source_only` branch). Selection state, the `Import selected (N)` POST body, and every wire-bound identity check use this value as-is. Rows with `presence === "stored_only"` carry `sourceSessionKey === null` and are not selectable.
- **React row identity** (`SessionRow.rowKey`, `apps/frontend/src/features/sessions/types.ts:24-25`) defaults to the same backend `session_key` for source-backed rows and falls back to `stored:${session_uid}` for `stored_only` rows (`apps/frontend/src/features/sessions/mergeSessions.ts:122`). This fallback exists ONLY for React keys and the drawer-open lookup; it NEVER enters an HTTP request body. The leading `stored:` prefix is intentional so a future audit can immediately tell a React-only fallback apart from the backend-issued import key.

### Refreshing the bindings

The regenerator is a normal integration test that is marked `#[ignore]` so it never runs in ordinary test invocations (it is the only test that writes to `bindings/`). To refresh the checked-in output after changing any contract type:

```bash
cargo test -p distill-portal-ui-api-contracts --features ts-bindings -- --ignored regenerate_ts_bindings
```

Commit the updated `.ts` files together with the Rust change in the same commit.

### Verifying the bindings are fresh

The default ts-bindings test run (without `--ignored`) regenerates into a temporary directory and compares against the checked-in files, failing if any byte differs:

```bash
cargo test -p distill-portal-ui-api-contracts --features ts-bindings
```

Run this locally before committing a contract change to confirm the TS output was regenerated. The test panic message includes the exact remediation command.

### What the generator does not do

- It does **not** change any `#[serde(...)]` attribute or wire shape. Adding or removing a `#[ts(...)]` attribute only affects the generated TS and never the JSON the backend emits.
- It does **not** automatically rewrite `bindings/` during `cargo test --features ts-bindings`; only the explicit `--ignored regenerate_ts_bindings` invocation writes to disk. The staleness test uses a temp dir so the verification step never silently "fixes" drift.
